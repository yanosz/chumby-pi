Date: 2026-07-07. Implements the Big Cleanup CHECKPOINT BC3 (plan
§"The Big Cleanup"): GitHub Actions green on yanosz/chumby-ruffle and

## 1. Common design

- **controlpanel.swf in CI**: fetched from the private Nextcloud share
  (dedicated `chumby-ci` user, doc 14 §7.3) with rclone configured
  purely via environment variables. `RCLONE_CONFIG_RSHARE_{TYPE,VENDOR}`
  are plain workflow values (`webdav`/`nextcloud` — reveal nothing);
  `RCLONE_CONFIG_RSHARE_{URL,USER,PASS}` are GitHub Actions secrets on
  both repos, so the server endpoint never appears in either repo.
  `PASS` holds rclone's *obscured* password form, exactly as in the
  local `rclone.conf` (config-style env vars take config-file values).
- **"Movie starts" assertion** (both repos, the BC1/BC3 acceptance
  "builds and controlpanel.swf runs"): run the player headless
  (`xvfb-run`, mesa lavapipe/llvmpipe) with
  `RUST_LOG=warn,chumby_host=info,avm_trace=info` under
  `timeout 45`; success = exit code **124** (player still alive when
  the timeout fired — no crash) AND `_getPlatform` present in the log
  (the panel executed a chumby ASnative call) AND no `panicked` line.
  Audio-device failure on a headless runner is non-fatal (verified
  locally, and already known from desktop runs).
- **Fixtures without copyrighted content**: verified locally
  2026-07-07 — the panel boots from the *public* fixture tree; the
  gitignored widget SWFs only cause a non-fatal `FetchError` on
  widget load. So CI needs no copyrighted material beyond the fetched
  controlpanel.swf.
- **Copyright containment**: the SWF is never committed, never cached,
  never uploaded. On chumby-pi it ends up inside
  `chumby-player-data_*.deb`, which is therefore excluded from
  artifacts — only the SWF-free `chumby-player` deb is uploaded.

## 2. chumby-ruffle: `.github/workflows/chumby.yml`

Triggers: push/PR on `chumby` (the default branch) + manual dispatch.
One job on `ubuntu-24.04`:

1. `cargo build -p ruffle_desktop --features chumby` (debug profile —
   the build-passes check; rust-cache for speed).
2. rclone-fetch controlpanel.swf + `fixtures.tar.gz` (see §5) from the
   share — no GitHub cross-repo access at all.
3. Headless movie-start assertion (§1).

Inherited upstream workflows were left untouched (minimizes future
upstream-merge conflicts): their push/PR triggers filter on `master`
(inert on `chumby`), and the scheduled ones (release nightly,
translations, extension test) are guarded by
`github.repository == 'ruffle-rs/ruffle'` — the 2026-07-07 nightly
Release run correctly self-skipped. Dependabot however IS active on
the repo (repo setting, not a workflow) and opened update PRs; Jan
may want to disable it in Settings → Code security.

## 3. chumby-pi: `.github/workflows/ci.yml`

Triggers: push/PR on `main` + manual dispatch. One job on
**`ubuntu-24.04-arm`** (GitHub-hosted arm64, the BC3 decision —
no qemu needed, no self-hosted runner):

1. Checkout with `submodules: recursive` (`ruffle/` = the fork over
   HTTPS, pinned by the gitlink; bumped to `05e6e698c` with this
   change).
2. `cargo build -p ruffle_desktop --features chumby --profile dist
   --target aarch64-unknown-linux-gnu` — native arm64 build, but
   `--target` keeps the output at the exact path `pkg/build-debs.sh`
   expects (same as the doc-08 cross-build). `dist` = what we ship
   (doc 11/12); the fat-LTO link is slow but the fidelity is the
   point.
3. rclone-fetch the SWF into `swf-assets/` (the build-debs.sh
   default), `pkg/build-debs.sh` → 0.2.0 debs.
4. **Install/run test** in an arm64 `debian:trixie` container
   (Raspberry Pi OS trixie's base) via
   `.github/scripts/deb-install-test.sh`: `apt-get install` of both
   debs (resolves every declared Depends — this is what catches
   undeclared deps), `ruffle_desktop --version`, then the §1
   movie-start assertion from the *packaged* SWF + fixtures.
5. Upload `chumby-player_*.deb` as artifact; the data deb stays in CI.

## 4b. Repo visibility + CI_REPO_TOKEN (2026-07-07, user decisions)

Both repos started out **private**, which the first chumby-pi run
(28847886126, failed in 12 s) exposed: the default `GITHUB_TOKEN` is
scoped to its own repo, so the checkout could not clone the
chumby-ruffle submodule ("Repository not found"), and the fork's
fixtures clone of chumby-pi fails the same way. Jan's final call:
**chumby-ruffle public** (flipped by Jan in the web UI — the CI
account jluehr-4711 has push but not admin rights, so Claude cannot
change visibility), **chumby-pi stays private**. Consequences:
- chumby-pi's submodule clone needs no token (submodule is public).
- The fork's fixtures clone still needs a fine-grained PAT (resource
  owner yanosz; repository access: chumby-pi only; permission
  Contents: read-only) stored as the `CI_REPO_TOKEN` secret on
  **chumby-ruffle**, passed as `token:` to actions/checkout.
- `ubuntu-24.04-arm` works on the private repo (the 12 s run did
  start), but chumby-pi's Actions minutes are billed against the
  account's free quota and the dist build is a long job — keep an eye
  on the quota.
- Public-fork hygiene: scheduled upstream workflows self-skip (§2) and
  nothing secret is in the tree; the `CI_REPO_TOKEN`/rclone secrets
  are repo secrets, not readable from fork PRs.

## 5. Secrets provisioning (Jan)

`gh secret set` and rclone *uploads* are blocked for Claude (auto-mode
classifier, same family as [push-to-main]); Jan runs, from the dev
box. Needed once per repo — `PASS` was set by Jan (chumby-pi
2026-07-06, chumby-ruffle 2026-07-07), leaving URL and USER:

```sh
for repo in yanosz/chumby-ruffle yanosz/chumby-pi; do
  awk -F' = ' '$1=="url"{print $2}'  ~/.config/rclone/rclone.conf \
    | gh secret set RCLONE_CONFIG_RSHARE_URL  -R "$repo"
  awk -F' = ' '$1=="user"{print $2}' ~/.config/rclone/rclone.conf \
    | gh secret set RCLONE_CONFIG_RSHARE_USER -R "$repo"
done
```

And the fixtures tarball upload — repeat this **whenever `fixtures/`
changes**, since the fork CI consumes the share copy, not the repo:

```sh
git -C /home/jan/chumby-pi archive HEAD fixtures | gzip \
  | rclone rcat rshare:testdata/fixtures.tar.gz
```

(§4b's CI_REPO_TOKEN PAT became unnecessary before it was ever
created: with chumby-ruffle public the submodule clone is anonymous,
and the fork CI now gets fixtures from the share instead of from the
private chumby-pi repo.)
